## 상황

리뷰 분석 기능을 만들면서 "시술 카테고리별 만족도"를 집계하는 쿼리를 짰다.
테이블 6개를 조인하는 꽤 무거운 쿼리다. 코드는 돌아가는데, 문득 이런 게 궁금해졌다.

- 이 쿼리, 실제로 빠른 건가 느린 건가?
- 인덱스가 왜 필요하다는 거지? 아무 컬럼에나 다 걸면 되는 거 아닌가?
- "빠르다/느리다"를 감이 아니라 **눈으로 확인**할 방법은 없나?

프론트를 주로 하다 보니 DB 실행 계획은 늘 남의 얘기였다. 이번엔 실제 코드와 DB로
하나씩 직접 확인해봤다.

## 해결 과정

### 1. 인덱스 = 책 뒤 "찾아보기"

500쪽 책에서 특정 단어를 찾는다고 하면:

- **인덱스 없음**: 1쪽부터 끝까지 다 넘긴다 → DB에선 `Seq Scan`(전체 훑기)
- **인덱스 있음**: 책 뒤 색인 보고 해당 쪽으로 바로 점프 → DB에선 `Index Scan`

즉 인덱스는 **자주 찾는/거르는 컬럼에 미리 만들어둔 색인**이다. 내 집계 쿼리는 항상
리뷰 날짜로 기간을 거르는데, 그 날짜 컬럼엔 이미 인덱스가 걸려 있었다.

```python
Index("ix_review_details_review_date", text("review_date DESC"))
```

### 2. 그럼 모든 컬럼에 인덱스를 걸면 되지 않나?

아니었다. 인덱스는 공짜가 아니다.

- 책에 문장을 새로 추가할 때마다 **뒤 색인도 매번 갱신**해야 한다.
- 인덱스가 10개면, 데이터 하나 넣을 때 색인 10개를 전부 고친다 → **저장(INSERT)이 느려진다.**

인덱스는 **조회는 빠르게 / 저장은 느리게**의 맞바꿈이다. 특히 리뷰처럼 수집할 때 대량으로
쏟아져 들어오는 테이블은 인덱스를 남발하면 수집이 느려진다. 그래서 "자주 조회·조인하는
컬럼에만" 신중하게 단다.

### 3. 언제 인덱스를 "일부러 안 다는가"

이번에 백필(과거 데이터를 나중에 채우는 작업)용으로 마커 컬럼을 하나 추가했다.
이 컬럼으로 필터를 거는데도, **인덱스를 일부러 안 달았다.**

판단 기준은 "필터 컬럼이냐"가 아니라 **"이 쿼리가 얼마나 자주 도나"** 였다.

- 백필 = 가끔, 필요할 때만 도는 배치 작업
- 사용자가 화면 볼 때마다 도는 쿼리가 아님

가끔 도는 쿼리 하나 빠르게 하자고, 데이터 저장할 때마다 인덱스 갱신 비용을 내는 건
손해다. 그래서 안 달았다. ("필터가 있으면 무조건 인덱스"가 아니라는 걸 처음 체감했다.)

### 4. EXPLAIN ANALYZE로 실제 계획 보기

쿼리 앞에 `EXPLAIN`을 붙이면 DB가 "나 이렇게 실행할 계획이야"를 보여준다.
`ANALYZE`까지 붙이면 **실제로 돌려보고** 걸린 시간과 처리한 행수까지 알려준다.
(단, `ANALYZE`는 진짜 실행하므로 조회에만 쓴다.)

단일 테이블에 먼저 돌려봤다.

```
Bitmap Heap Scan on review_details  (rows 22950 예상) (actual rows=23560)
  -> Bitmap Index Scan on ix_review_details_review_date
        Index Cond: (review_date >= '2026-01-01')
 Execution Time: 12.093 ms
```

읽는 법:

- `Bitmap Index Scan on ix_..._review_date` → **날짜 인덱스를 실제로 탔다.** 6만 건 전체를
  안 훑고 색인으로 해당 구간만 집었다는 증거.
- `Bitmap`은 `Seq`와 `Index`의 중간 형태다. 매칭이 아주 적으면 `Index Scan`, 거의 전부면
  `Seq Scan`, 이번처럼 **어중간하게 많으면(6만 중 2.3만)** DB가 `Bitmap`을 고른다.
- `rows=22950`(예상) vs `actual rows=23560`(실제)가 거의 같다 → DB 통계가 건강하다는 신호.
- 2.3만 건 집는 데 **12ms**. 인덱스 덕분이다.

### 5. 6개 조인 쿼리는 어떻게 실행될까

본 게임. 6개 테이블을 조인하는 집계 쿼리를 `EXPLAIN ANALYZE`로 돌리니, 여러 줄짜리
트리가 나왔다. **실행 계획은 들여쓰기가 깊은 안쪽부터 읽는다.** 핵심만 추리면:

```
1. treatment_tags 읽기 (1,066건)      <- 여기서 출발!
2. + analyses 랑 Hash Join
3. 각 건마다 -> analysis 찾기  (Index Scan, loops=1066)
4. 각 건마다 -> review_details 찾기 (Index Scan, 날짜필터)
5. 각 건마다 -> reviews 찾기       (Index Scan)
6. + source_configs 붙이기 (Seq Scan, 106건)
7. category_key로 묶고 개수순 정렬
= 67ms
```

여기서 제일 인상 깊었던 3가지.

**(1) DB가 제일 작은 표부터 시작했다.**
리뷰는 6만 건, 시술 태그는 1,066건. DB는 6만 건이 아니라 **1,066건 태그부터** 조인을
시작했다. 시술 태그가 있는 리뷰는 어차피 1,066건뿐이라, 그것부터 붙잡고 나머지를 찾는 게
헛수고가 적다. 이걸 **조인 순서(join order)** 라 하고, DB가 데이터 크기(selectivity)를 보고
**스스로** 정한다.

**(2) 1,066번 반복해도 빠르다 — 매번 인덱스 점프라서.**

```
Index Scan using analyses_pkey ... (loops=1066)
```

- `loops=1066` = 이 찾기를 1,066번 반복
- `Index Scan using ..._pkey` = 매번 기본키 인덱스로 점프

즉 "1,066번 반복했지만 매번 색인 점프라 빨랐다." 인덱스 없이 매번 6만 건을 훑었다면
1,066 × 6만이 됐을 것이다.

**(3) 작은 표는 Seq Scan이 정답.**
`source_configs`(106건)는 인덱스 대신 `Seq Scan`으로 통째로 읽었다. 1쪽짜리 쪽지는 색인
찾는 것보다 그냥 눈으로 읽는 게 빠른 것과 같다. **Seq Scan이 무조건 나쁜 게 아니라,
"큰 테이블에서 Seq Scan이 나올 때"만 의심하면 된다.**

결과적으로 6개 조인 집계가 인덱스 + 병렬 처리로 **67ms**에 끝났다.

## 배운 것

| 개념 | 한 줄 |
|---|---|
| 인덱스 | 책 뒤 색인. 자주 찾는 컬럼에 만들면 전체 안 훑고 점프 |
| 인덱스의 대가 | 조회는 빨라지지만 저장이 느려짐 → 필요한 것만 |
| 안 다는 판단 | "자주 도는 쿼리냐"로 결정. 가끔 도는 배치엔 안 달 수도 |
| EXPLAIN | 실행 전 계획서. Seq Scan vs Index Scan |
| EXPLAIN ANALYZE | 실제 돌린 실측. 예상 vs 실제 행수, 실제 시간 |
| 조인 순서 | 작은 표부터 시작 = 헛수고 적음. DB가 알아서 고름 |
| loops + Index Scan | 여러 번 반복해도 매번 색인 점프면 빠름 |
| 작은 표 Seq Scan | 꼬마 표는 통째로 읽는 게 정답. Seq Scan이 늘 나쁜 건 아님 |

가장 크게 남은 건 **"감이 아니라 EXPLAIN ANALYZE로 확인한다"** 는 태도다.
인덱스를 걸지 말지, 이 쿼리가 빠른지 느린지를 추측하지 않고, 실제 실행 계획을 열어
`Seq Scan`인지 `Index Scan`인지, 예상과 실제 행수가 맞는지, 어느 단계가 오래 걸리는지를
눈으로 본다. 프론트만 하던 입장에서 DB가 한결 덜 무서워졌다.
