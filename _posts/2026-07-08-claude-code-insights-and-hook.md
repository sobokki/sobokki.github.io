## 상황

Claude Code를 한 달 넘게 쓰다 보니 문득 궁금해졌다. "나는 이 도구를 대체 어떻게
쓰고 있지? 반복해서 삽질하는 지점은 어디지?" 그때 써본 게 `/insights`다.

이 글은 세 갈래로 정리한다. **(1) `/insights`가 뭔지, (2) 실행하면 어떤 식으로
정보가 오는지, (3) 나는 그걸 어떻게 적용해 실제로 뭘 개선했는지.**

## 1. `/insights`가 뭔가

`/insights`는 Claude Code에 내장된 **내 사용 패턴 분석 기능**이다. 최근 세션들을
훑어서 이런 걸 정리해준다.

- 내가 **주로 어떤 작업**에 이 도구를 쓰는지 (코드 편집, 문서화, git 작업 등)
- **반복되는 마찰 지점** — 어디서 자주 되돌리고 다시 시키는지
- 자주 쓰는 도구·언어, 시간대별 사용량, 추정 만족도
- 그리고 위 분석에 기반한 **개선 제안**

주의할 점 하나: 분석 대상은 "지금 이 폴더"가 아니라 **이 컴퓨터에서 작업한 전체
세션**이다. 실행 위치와 무관하게 로컬에 쌓인 기록 전부를 본다.

## 2. 실행하면 정보가 어떻게 오나

결과는 **HTML 리포트 한 장**으로 나온다. 단순 통계 나열이 아니라, 섹션으로 나뉘어
읽히게 구성돼 있다.

- **한눈에 보기** — 잘 되는 점 / 발목 잡는 점 / 바로 시도할 개선 요약
- **작업 영역** — 최근 몇 주간 뭘 했는지 묶어서
- **사용 방식 서술** — "당신은 이런 식으로 일한다"는 문장형 분석
- **마찰 지점** — 어떤 유형의 오류가 반복됐는지 + 구체 예시
- **시도해볼 기능 / 개선 제안** — 여기가 핵심

리포트 안에는 아래처럼 **차트로 된 사용 통계**도 들어있다. (제품 정보가 없는 중립
지표만 추린 화면이다.)

![Claude Code /insights 사용 통계 차트](/assets/img/posts/insights-charts.png)
_리포트가 보여주는 사용 통계 예시 — 도구 사용량, 세션 유형, 응답 시간, 도구 오류_

특히 개선 제안은 말로만 끝나지 않고, **규칙 문서에 바로 붙여넣을 수 있는 스니펫**과
**복사용 프롬프트**까지 준다. "이렇게 하면 좋다"가 아니라 "이 문장을 여기에 넣어라"
수준으로 구체적이다. 즉 전달 형식 자체가 **"읽고 끝"이 아니라 "바로 적용"**을 노린다.

## 3. 나는 이걸 어떻게 적용해 개선했나

제안을 받았다고 그대로 다 넣지는 않았다. 오히려 여기서 진짜 배운 게 나왔다.

### (1) 제안은 기존 규칙과 대조부터

받은 제안 5개 중 절반은 **이미 내 규칙 문서에 있던 내용**이었다. 그대로 추가하면
같은 규칙이 두 곳에 생겨 나중에 충돌·혼란을 부른다. 그래서 기존 규칙과 대조하고
**진짜 빠져 있던 것만** 반영했다.

### (2) "편집 후 자동 lint 훅"을 넣으려다 이상한 걸 발견

제안 중 하나가 "파일 편집 후 자동으로 lint를 돌리는 훅을 걸어라"였다. 우리
레포(FE/BE 모노레포)엔 이미 **백엔드용 자동 포맷 훅**이 있어서 프론트용만 추가하려
했는데 — 기존 훅이 편집 파일 경로를 `$file_path`라는 **환경변수**로 읽고 있었다.

```bash
# 기존 훅 — 파이썬 파일이면 포맷터 실행
if [[ "$file_path" == *.py ]]; then run_formatter "$file_path"; fi
```

이게 진짜 되는 변수인지 의심스러워 공식 문서를 확인했다. 결과는 예상 밖이었다.

**Claude Code 훅은 편집 경로를 환경변수로 주지 않는다.** 데이터는 **stdin에 JSON**
으로만 오고, 경로는 `.tool_input.file_path`에 있다. 즉 `$file_path`는 애초에
**존재하지 않는 변수**였다.

### (3) 재현으로 "무동작"을 확인

말로 판단하지 않고 직접 재현했다.

```bash
# $file_path 환경변수는 비어있다
echo '{"tool_input":{"file_path":"/x/y.py"}}' \
  | sh -c 'if [[ "$file_path" == *.py ]]; then echo "실행됨"; else echo "무동작: [$file_path]"; fi'
# 출력: 무동작: []
```

조건이 늘 `[[ "" == *.py ]]` = 거짓. **훅은 설정돼 있었지만 한 번도 실행된 적이
없었다.** "설정이 존재한다"와 "동작한다"는 다른 얘기였다.

그래서 `jq`로 stdin에서 경로를 뽑는 방식으로 새 훅을 만들었다.

```bash
# stdin JSON에서 경로 추출 → 실제로 동작
FILE=$(jq -r '.tool_input.file_path // empty')
if [[ "$FILE" == *.ts || "$FILE" == *.tsx ]]; then
  run_linter --fix "$FILE"
fi
```

```bash
# 재현으로 동작 확인
echo '{"tool_input":{"file_path":"/path/App.tsx"}}' \
  | sh -c 'FILE=$(jq -r ".tool_input.file_path // empty"); [[ "$FILE" == *.tsx ]] && echo "OK: $FILE"'
# 출력: OK: /path/App.tsx
```

### (4) 팀 공유냐 개인이냐, 위치부터 정한다

훅을 넣기 전에 하나 더. 프로젝트 설정 파일은 **git으로 추적돼 팀 전체에 적용**되고,
`*.local` 류의 개인 설정은 무시(gitignore)된다. "팀에 강제할 규칙"인지 "나만 써볼
것"인지에 따라 넣는 위치가 달라진다. 이걸 먼저 정하고 넣었다.

## 배운 것

- **`/insights`는 통계가 아니라 실행 가능한 제안을 준다.** 다만 그대로 붙이지 말고
  기존 규칙과 대조부터. 중복 규칙은 지금 편하자고 나중에 충돌을 만든다.
- **추측 대신 재현으로 검증한다.** "설정 파일에 있다 ≠ 동작한다." 30초짜리 재현
  커맨드 하나가 한참의 오해를 끝냈다.
- **도구의 입력 인터페이스는 공식 문서로 확인한다.** 관성으로 쓰던 변수명이 사실
  아무 값도 아닐 수 있다.
- **설정은 위치가 곧 범위다.** 팀 공유(git 추적) vs 개인(gitignore)을 먼저 구분한다.
