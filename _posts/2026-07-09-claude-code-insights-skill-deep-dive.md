영상은 두 부분이다. 전반부는 "insight 스킬이 뭘 해주냐", 후반부는 "그게 어떻게

## 1. insight 스킬이 하는 일

한 줄로: **내 Claude Code 세션 히스토리를 분석해서 개인화된 코딩 리포트(HTML)를
로컬에 만들어준다.** 단순 사용량 대시보드가 아니라는 게 핵심이다.

리포트에 담기는 것:

- 주요 작업 영역 5개, 각 영역별 세션 수
- 잘 되고 있는 것 / 마찰 포인트(개선이 필요한 지점)
- **바로 복붙할 수 있는 개선안** — CLAUDE.md에 추가할 내용, 만들면 좋을 커스텀 스킬,
  추가할 pre-commit hook, 써볼 만한 MCP까지

그러니까 "너 이렇게 쓰고 있어"에서 끝나는 게 아니라 "이렇게 바꿔봐"까지 준다.
영상에서는 이걸 **자기 개선 루프**라고 표현했다. 리포트가 주는 프롬프트를 실제로
적용하면 그만큼 Claude Code가 내게 맞춰진다는 것.

## 2. 내부 동작 — 6단계 파이프라인

여기가 핵심이다. insight 스킬은 이렇게 돈다.

| 단계 | 하는 일 |
|------|---------|
| 1. Collection | 어떤 세션이 있는지 수집 |
| 2. Filtering | 조건 안 맞는 세션 걸러냄 |
| 3. Metadata | 세션 ID·지속시간·토큰·툴 호출·언어·git 활동 추출 |
| 4. Summarize | 세션 요약 (긴 건 청킹) |
| 5. Facets 추출 | 세션의 "단면"을 뽑아 캐싱 |
| 6. Aggregate → Report | 종합 분석 후 HTML 리포트 생성 |

세부 규칙 중 기억해둘 것:

- **100% 로컬 실행.** 세션 내용을 웹으로 보내지 않는다. 분석·리포트·캐시 전부 로컬.
  (프라이버시 때문)
- **모델은 Haiku.** 50개 세션을 각각 요약하고 facet 뽑고 다시 종합하려면 토큰이
  많이 드는데, Opus를 쓰면 비용 손해다. 고추론이 필요 없는 추출 작업이라 Haiku가 맞다.
- **필터 조건**: 사용자 메시지 2개 이상 + 1분 이상 지속된 세션만. 서브에이전트
  세션은 제외(너무 방대하고 인사이트가 적음). 1회 실행당 **최대 50개 세션**.
- **청킹**: 세션이 3만 자를 넘으면 25,000자 단위로 쪼갠다. Haiku 컨텍스트를
  효율적으로 쓰기 위한 것.
- **증분 캐싱**: facet을 `usage-data/facets/<세션ID>.json`으로 저장해두고,
  이미 분석한 세션은 다시 분석하지 않는다. 반복 실행 비용을 최소화.

`facet`이라는 개념이 좋았다. 세션 하나에서 뽑는 네 가지 단면이다.

- **Goal** — 이 대화의 목적 (예: React 컴포넌트를 서버 컴포넌트로 마이그레이션)
- **Satisfaction** — 좋았던 점 (피드백 없이 바로 수락됨 = 좋은 신호)
- **Friction** — 마찰점 (몇 번씩 다시 시킨 지점)
- **Outcome** — 성공/실패

세션 데이터 원본은 `~/.claude/projects/` 아래에 `.jsonl`로 쌓이고, 리포트는
`~/.claude` 아래 HTML로 떨어진다.

## 3. ⚠️ 알아야 할 한계와 버그

영상에서 제일 강조한 부분이고, 나도 이게 제일 중요하다고 느꼈다.

- **소량 데이터 환각.** 세션이 10개 미만으로 적으면, 집계 단계에서 Haiku가
  "336개 세션" 같은 숫자를 **지어낸다.** facet 데이터가 부족하면 그럴듯한 숫자를
  만들어버리는 것. 리포트를 억지로 채우려다 생기는 문제로 보인다.
- **50개 세션 제한** — 그 이상 쓰면 실행 주기를 당겨서 나눠 돌려야 한다.
- **분석 기간 30일 고정** — "이전 30일"처럼 구간 지정도 안 된다. (1년치를 넣으면
  컨텍스트가 터지니까 어쩔 수 없이 끊어둔 것)
- **HTML만 출력** — JSON/CSV 없음, MCP 연결도 어려움.
- **런타임** — 600개 메시지 규모면 15분쯤 걸린 적도 있다고.

그래서 저자는 매일 돌리지 말고 **2주에 1번** 정도를 권했다. 나도 세션이 아직
많지 않아서, 환각 이슈를 피하려면 어느 정도 쌓인 다음에 돌리는 게 낫겠다 싶었다.

## 정리

insight를 표면만 볼 땐 "예쁜 사용량 리포트"로 보였는데, 뜯어보니 **세션 → 요약 →
facet → 집계**로 이어지는 작은 분석 엔진이었다. 특히:

- Haiku를 추출용으로 쓰고 캐싱으로 재실행 비용을 줄이는 **모델 역할 분담**
- 세션의 질적 단면을 뽑는 **facet** 설계
- 그리고 데이터가 적을 때의 **환각**이라는 실전 함정

이 세 개가 남는다. 다음엔 세션을 좀 더 쌓은 뒤 다시 돌려서, 리포트가 주는 개선
프롬프트를 실제 CLAUDE.md에 적용해보고 후속 글을 써볼 생각이다.
