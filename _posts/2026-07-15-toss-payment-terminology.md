> 결제 연동에서 제일 어려운 건 API가 아니라 **용어와 정책**이었다.

## 상황

우리 서비스는 월 구독에 크레딧 충전을 얹는 구조다. 구독(자동결제)은 이미 토스페이먼츠 연동과 심사를 마쳤고, 남은 건 **충전 — 필요할 때 원하는 금액을 결제하는 단건결제**였다.

회사에는 여러 제품이 함께 쓰는 **사내 결제 서버**가 있다. 각 제품은 토스를 직접 부르지 않고 이 서버를 거친다.

```
제품 → 사내 결제 서버 → 토스페이먼츠
```

사내 가이드의 단건결제는 간단해 보였다. *"등록해둔 카드에서 한 번 긁는 결제. 서버가 저장된 빌링키로 즉시 청구한다."* 가이드대로 구현하고, 충전용 상점 키도 발급받았다.

그런데 카드 등록 창을 여는 순간 토스가 거절했다.

> **"자동 결제(빌링) 계약이 안 되어 있습니다."**

## 혼란: 같은 단어, 다른 사전

기획자에게 물으니 답이 왔다. *"그 상점은 **일반결제** 계약이에요."*

여기서 혼란이 시작됐다. "일반결제"의 뜻이 화자마다 달랐다.

- **토스의 일반결제** = 결제할 때마다 결제창이 뜨는 방식
- **사내 가이드의 일반결제** = 창 없이 저장된 카드로 긁는 방식

우리는 토스 쪽 뜻으로 알아듣고 결제창 방식으로 갈아탔다. 그런데 결제 서버 코드를 열어보니 **결제창을 받아줄 API 자체가 없었다.** 서버는 처음부터 "모든 결제는 빌링키로"라는 전제로 만들어져 있었다. 구현은 이렇게 오락가락했다:

```
카드 재등록 방식 → 결제창 방식 → 구독 카드로 청구 → ???
```

## 전환점: 문서에 없던 토스 정책

감으로 싸울 게 아니라 근거가 필요했다. 토스 공식 문서에서 이 문장을 찾았다.

> "**카드사에서 빌링키의 비정기적인 결제를 허용하지 않습니다.** 자동결제는 정기 구독형 서비스 전용입니다."

이 한 줄로 퍼즐이 풀렸다. 충전은 "비정기 결제"다. 그러니 **빌링키(저장 카드)로 충전을 긁는 것 자체가 카드사 규정 밖**이었고, 토스가 충전용 상점에 빌링 계약을 안 붙여준 것도, 상점을 구독용/단건용으로 나눠준 것도 전부 이 정책 때문이었다.

그리고 진짜 문제가 보였다. **사내 결제 서버가 이 정책을 모른 채 설계돼 있었다.** "모든 결제 = 빌링키"라는 전제는 구독엔 맞지만 충전엔 규정 위반이었고, 이 사실은 사내 문서 어디에도 없었다.

## 어떻게 확신했나: 문서 → 코드 → 실측

- **문서 대조** — 토스 문서와 사내 가이드·기획서를 나란히 놓고 모순 지점을 표로 정리
- **코드 확인** — 결제 서버가 토스에 부르는 API를 전수 확인 → 전부 빌링키 계열, 결제창 승인 없음 (추측 종료)
- **실측** — 스테이징에서 에러 원문을 직접 받아내고, 발급받은 키들이 어느 계층의 자격증명인지 분석

증거가 쌓이자 논쟁이 사라졌다.

## 결말: 인프라가 고쳐졌고, 모든 제품이 방향을 바꿨다

증거를 기획자와 결제 인프라 담당자에게 전달했고, 진단이 하나로 모였다.

> 원인은 **토스 정책을 반영하지 않은 사내 결제 서버.** 토스에서 결제는 상점(계약)별로 분리되어야 하고, 충전은 단건 상점 키로 별도 연동해야 한다 — **결제 서버를 수정한다.**

이게 이 글의 핵심이다. 결제 서버는 **전사 공용 인프라**다. 우리가 한 제품에서 이 충돌을 일찍 발견하지 못했다면, 다른 제품들도 같은 전제 위에 충전을 쌓아 올리다 심사·운영 단계에서 훨씬 비싸게 부딪혔을 것이다. **개발이 더 진행되기 전에 인프라의 방향이 교정됐고, 결제가 붙는 모든 제품의 일정이 올바른 전제 위에 다시 놓였다.**

## 배운 것

1. **용어 말고 행동으로 물어라.** "일반결제 맞아요?"보다 "결제할 때 창이 뜨나요?"가 정확하다.
2. **PG 정책이 API보다 위에 있다.** 호출이 성공해도 카드사 규정이 거부권을 가진다.
3. **추측이 갈리면 코드와 실측으로 끝내라.** grep 한 번, 스테이징 요청 한 번이 회의 세 바퀴보다 빠르다.
4. **접은 구현은 지우지 말고 보존하라.** 방향이 뒤집혀도 전환 비용이 하루를 넘지 않았다.
5. **공용 인프라의 구멍은 발견한 사람이 전파해야 한다.** 그 한 번의 이슈 제기가 다른 팀들의 몇 주를 구한다.
