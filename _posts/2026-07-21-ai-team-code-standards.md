> AI 협업의 코드 품질을 결정하는 건 프롬프트가 아니라, **리포지토리에 박아둔 컨텍스트**였다.

## 상황 — 리뷰가 따라잡을 수 없는 속도

병원 CRM을 만들던 팀의 구조는 특이했다. 디자이너 없이, **도메인별로 각 담당자가 디자인 구현부터 프론트·백엔드까지** 맡는 풀스택 체제. 그리고 팀 전체가 Claude Code로 개발하는, AI 협업 비중이 큰 팀이었다.

프론트는 6개 도메인(현황판·고객·예약·차트·문서·경영), 라우트 15개 규모로 커질 예정이었다. 시작 전에 이 조합이 만들 불일치를 예측해 봤다.

- 같은 버튼도 만드는 **사람마다** 색·구조·파일 위치가 달라질 것이다.
- 같은 사람이라도 **AI 세션마다** 결과가 달라질 것이다. LLM은 확률적이라, 어제의 세션이 지킨 컨벤션을 오늘의 세션은 모른다.
- 백엔드 담당자가 프론트 코드를 만질 일이 구조적으로 잦을 것이다 — 프론트 컨벤션을 모르는 채로.

이 세 가지는 발생한 뒤에 리뷰로 잡을 수 있는 성질이 아니다. 화면이 늘어나는 속도가 리뷰 처리 속도를 넘을 것이 뻔했고, 무엇보다 리뷰는 이미 발생한 불일치를 **발견**하는 도구지 발생 자체를 **막는** 도구가 아니다. 불일치는 한 번 리포지토리에 들어오면 복사되며 퍼지기 때문에, 사후 교정은 시작부터 지는 게임이었다.

그래서 화면을 만들기 전에, 일관성을 **구조로 먼저 고정**하기로 했다. 프론트 리포지토리의 `.claude/` 디렉토리 전체를 첫 화면이 나오기 전에 직접 설계했다 — **룰 문서 6종, 스킬 5개, 검증 훅 3개, 커맨드 3개, 역할 에이전트 2개.** 사람이 짜든 AI가 짜든 같은 결과가 나오는 환경을 먼저 깔고, 그 위에서 화면을 만드는 순서였다.

```text
.claude/
├── rules/      # 6종 — api · design · func · page · refactor · ui
├── skills/     # 5개 — 반복 작업 절차
├── hooks/      # 3개 — 결정적 자동 검증 (금지 패턴 · 파일 위치 · 라우트 등록)
├── commands/   # 3개 — 도메인 스캐폴딩 · 감사 · 프론트 리뷰
└── agents/     # 2개 — 화면 구현용 · 디자인 검증용
```

## 해결 과정

### 1. 기준부터 — 무엇을 고정할 것인가

도구를 만들기 전에, 흔들리면 안 되는 기준을 세 층으로 정의했다.

**① 디자인 토큰 1:1 매핑.** 색은 디자인 시스템 토큰의 값만 허용하고 임의 색을 금지했다. 디자인 파일의 색과 코드 토큰을 1:1로 매핑한 **색 토큰표**를 룰 문서(design)에 넣어, "디자인의 이 색 = 코드의 이 토큰"이 조회 가능하게 했다.

```tsx
// ❌ 금지 — 디자인과 어긋날 수 있는 통로
<div className="text-[#4A7AFF]">

// ✅ 허용 — 토큰만
<div className="text-primary-500">
```

토큰만 쓰면 디자인이 바뀌어도 토큰 정의 한 곳만 바뀐다. 임의 색을 하나라도 허용하면 그 지점부터 디자인과 구현이 따로 늙는다.

**② 컴포넌트 우선순위.** 사내 디자인 시스템(공용 라이브러리) 1순위 → 프로젝트 공통 컴포넌트 → 둘 다 없을 때만 확인 후 신규 생성. 같은 컴포넌트의 중복 구현을 금지했다. AI는 "없으면 만들어 버리는" 경향이 있어서, **탐색 순서를 명시하지 않으면 비슷한 컴포넌트가 계속 늘어난다.**

**③ 파일 책임 분리.** 데이터 페칭은 Page, UI·로컬 상태는 Client — 어떤 코드가 어느 파일에 가는지 기준표로 고정했다. "이 코드 어디에 둘까"라는 판단을 사람·세션의 재량에 맡기지 않는다.

### 2. 룰 6종 — AI가 코드를 짜기 전에 읽는 문서

기준을 관심사별 룰 문서로 나눴다: **api**(연동 규칙) · **design**(토큰 매핑) · **func**(함수 컨벤션) · **page**(페이지 구조) · **refactor**(수정 시 원칙) · **ui**(컴포넌트 규칙). Claude Code는 작업 시작 시 해당 룰을 읽고 시작한다.

룰을 쓰면서 배운 것 하나 — **"위반하면 어떻게 되는지"까지 적어야 규칙이 실행된다.** 우리 룰에는 "위반하는 코드는 리뷰에서 반려된다"를 명시했다. 사람에게는 규칙의 무게를 전달하고, AI에게는 "이 규칙은 무시 가능한 참고사항이 아니다"라는 신호가 된다.

### 3. 훅 3개 — 확률적인 AI를 결정적으로 보정하는 마지막 방어선

룰 문서만으로 충분하지 않다는 건 설계 시점부터 전제였다. LLM은 확률적이어서 긴 세션에서는 초반에 읽은 규칙이 흐려진다. 그래서 룰과 별개로 **결정적(deterministic)으로 실행되는 검증 훅**을 처음부터 함께 뒀다. 훅은 AI의 컨디션과 무관하게 항상 돈다.

| 검증 훅 | 잡는 것 |
|---|---|
| 금지 패턴 검사 | 임의 색값 등 룰이 금지한 패턴이 코드에 들어오는 순간 검출 |
| 파일 위치 검사 | 파일이 정해진 폴더 구조(콜로케이션 규칙)를 벗어나면 검출 |
| 라우트 등록 검사 | 새 화면을 만들고 라우트 등록을 빠뜨리는 실수 검출 |

셋 다 lint와 같은 철학이지만 실행 시점이 다르다. 커밋 후 CI가 아니라 **AI가 코드를 쓰는 순간** 피드백이 돌아온다. 불일치가 리포지토리에 들어오기 전에 잡히고, 고치는 것도 그 자리에서 AI가 한다. 사람이 위반을 발견하고 → 코멘트를 달고 → 수정을 기다리는 리뷰 루프가 통째로 사라지는 지점이다.

훅 3개의 대상 선정에도 기준이 있었다. **어기기 쉽고(빈도), 어기면 조용히 퍼지는(전파성)** 것부터. 임의 색은 한 번 들어오면 복사되며 퍼지고, 잘못된 파일 위치는 다음 사람의 탐색 비용이 되고, 라우트 누락은 "만들었는데 안 보이는" 화면을 만든다.

### 4. 스킬·커맨드 — 반복 절차의 표준화

규칙 준수가 해결됐다면, 다음은 반복 작업의 편차였다. **새 도메인 화면을 추가하는 절차**(폴더 구조, 라우트, 기본 파일 구성)를 커맨드로 만들어 누가 실행해도 같은 뼈대가 나오게 했고, 감사(audit)용·프론트 리뷰용 커맨드, 화면 구현용·디자인 검증용 에이전트를 뒀다.

핵심은 이거다: **"이렇게 해주세요"라고 매번 설명하는 대신, 절차 자체를 리포지토리에 코드처럼 넣는다.** 설명은 세션이 끝나면 사라지지만, 스킬과 커맨드는 버전 관리되며 남는다.

### 5. 사람에게도 같은 문서가 기준이 된다

예상 못 한 부수 효과가 있었다. AI를 위해 쓴 룰 문서가 **사람에게도 온보딩 문서가 됐다.** 백엔드 담당자가 프론트를 만질 때 "뭘 지켜야 하죠?"의 답이 한 곳에 있다. 규칙이 사람 머릿속이나 리뷰 코멘트에 흩어져 있을 때는 불가능했던 일이다.

결국 이 작업의 본질은 AI 도구 설정이 아니라 **팀의 암묵지를 버전 관리되는 명시적 구조로 바꾸는 것**이었고, AI는 그걸 강제로 하게 만든 계기였다.

한 가지 아쉬움도 남긴다. 이 체계의 효과를 정량 측정할 지표(리뷰 반려율, 컨벤션 위반 빈도 등)를 **도입 전에 설계해 두지 않아서**, 효과는 정성적으로만 말할 수 있다. 다음에 같은 걸 만든다면 측정부터 설계하고 들어갈 것이다 — 개선을 주장하려면 개선 전의 숫자가 필요하다.

## 배운 것

1. **일관성은 리뷰로 지키는 게 아니라 구조로 만드는 것이다.** 리뷰는 불일치의 발견이고, 구조는 불일치의 예방이다. 화면이 늘어나는 속도가 리뷰 처리량을 넘는 순간, 사후 교정은 진다.
2. **AI 협업의 품질은 세션 프롬프트가 아니라 리포지토리 컨텍스트가 결정한다.** 매 세션 반복하는 설명은 휘발되고, 룰·스킬·훅은 버전 관리되며 누적된다.
3. **확률적인 도구에는 결정적인 방어선이 필요하다.** 룰(읽고 따르는 것)과 훅(무조건 실행되는 것)은 역할이 다르다. 룰만 있으면 세션이 길어질수록 새고, 훅이 그 구멍을 막는다.
4. **자동화 대상은 빈도 × 전파성으로 고른다.** 자주 어기고, 어기면 조용히 퍼지는 것부터 훅으로 만든다.
5. **효과를 주장하려면 측정을 먼저 설계해야 한다.** 도입 전 지표가 없으면 개선은 정성적 주장에 머문다 — 이번의 가장 큰 아쉬움이자, 다음의 첫 번째 할 일.
