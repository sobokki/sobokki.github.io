## 상황

프론트 개발을 하다 보면 늘 걸리는 지점이 있다. **STG 배포 → QA 검증 → 반려 → 다시 수정**의
왕복. QA가 잡아주는 이슈 중 상당수는 "내가 로컬에서 조금만 꼼꼼히 눌러봤으면 발견했을"
것들이다. 그래서 목표가 명확했다.

> **QA에게 넘기기 전에, 로컬에서 내가 먼저 최대한 잡아서 QA 환류를 줄이자.**

마침 [agent-browser로 AI에게 브라우저 눈을 달아주는 영상](/posts/agent-browser-autonomous-qa/)을
보고 "이거다" 싶었다. AI가 실제 브라우저를 몰아 사람처럼 테스트하는 그 방식을, 내 업무
프론트엔드의 **배포 전 로컬 게이트**로 써보기로 했다.

## 해결 과정

### 1. 두 가지 테스트를 구분하는 것부터

먼저 헷갈리지 말아야 할 게 있었다. 내가 이미 쓰던 것과 새로 도입하려는 게 **목적이 다른**
별개라는 점.

| | 기존 E2E (Playwright) | 새로 도입 (agent-browser) |
|---|---|---|
| 방식 | 스크립트 (미리 짠 스텝) | 자율 (AI가 직접 판단해 클릭·입력) |
| 대상 | mock API (결정적 회귀) | 실제 백엔드 (진짜 동작) |
| 잡는 것 | "예전에 되던 게 안 깨졌나" | "새로 만든 게 실제로 되나" |

둘은 경쟁이 아니라 **역할 분담**이었다. 회귀는 스크립트가, 탐색적 실동작 검증은 자율 QA가.

그리고 이 둘은 **QA 팀의 검증과도 별개**다. QA는 이후 단계에서 자체 자동화와 수동 탐색으로
다시 본다. 정리하면 세 층이 각자 자리를 갖는다.

| 층 | 누가 | 언제 | 성격 |
|---|---|---|---|
| 스크립트 E2E | 개발자 | 항상 | 결정적 회귀 (탐색 안 함) |
| **자율 QA (이번에 도입)** | **개발자** | **넘기기 전** | **탐색적 · 실백엔드 · SPEC 대조** |
| QA 검증 | QA 팀 | 개발 이후 | 자체 자동화 + 수동 탐색 |

내가 손댄 건 가운데 층 — 스크립트 회귀와 QA 사이의 빈틈을, **개발자가 넘기기 전에 스스로
메우는** 자리다.

### 2. 그냥 "동작하나"가 아니라 "기획대로 됐나"

핵심 결정. 자율 QA가 단순히 "폼이 제출되나"만 보면 반쪽짜리다.

우리 팀은 **문서 우선(doc-first)** 으로 일한다. 기능을 코드로 짜기 전에, 그 기능이 어떻게
동작해야 하는지를 **기능 명세서(이하 SPEC)** 에 먼저 글로 정의한다. "등록 버튼을 누르면
이런 검증을 거쳐 이런 응답이 오고 목록에 이렇게 반영된다" 같은 **기대 동작**이 SPEC에 다
적혀 있다. 그러니까 SPEC은 개발의 출발점이자, 동시에 **"이렇게 동작해야 정답"이라는 기준**
이다.

그렇다면 자율 QA가 봐야 할 건 명확했다. "화면이 떴나"가 아니라 **"SPEC에 적힌 대로
구현됐나"**. 그래서 QA 스킬이 사람 대신 **그 SPEC 문서를 읽어서 기대 동작을 정답지로
뽑고**, 실제 화면 동작을 그 정답지와 한 줄씩 대조하도록 만들었다.

```
① SPEC 문서를 읽어 기대 동작(정답지)을 추출
  → ② 로컬 실제 스택(FE+BE) 기동
  → ③ AI가 로그인 → 등록 여정을 자율 실행
  → ④ 각 단계를 ①의 정답지와 대조
  → ⑤ 불일치 = 기획대로 안 된 것 = QA가 잡았을 이슈
  → ⑥ 테스트로 만든 데이터는 자동 정리
```

오해를 피하자면, 이건 **QA를 대체하는 게 아니다.** QA 팀도 수동으로 눌러보기만 하는 게
아니라 자체 자동화 테스트로 검증한다. 내가 만든 건 그 **앞단** — 개발자가 QA에 넘기기 전에
스스로 한 번 거르는 **pre-check**다. 예전 같으면 기획서를 옆에 띄워두고 직접 눌러보며 맞나
확인했을 그 과정을, SPEC을 정답지로 읽는 자율 QA가 배포 전에 미리 한 번 돌려주는 것이다.
"동작 확인"이 아니라 "**기획(SPEC) 준수 검증**"이고, 목적은 QA 대체가 아니라 **QA까지
흘러가는 이슈를 줄이는 것**이다.

### 3. 파일럿 — 로그인 + 등록 여정

먼저 스킬부터 짜지 않았다. **한 번 손으로 돌려서 실제로 되는지** 확인하는 게 먼저였다.
(추측으로 만든 워크플로우는 십중팔구 틀리니까.)

로그인 가드가 첫 관문이었다. 공개 페이지가 아니라 인증을 통과해야 화면에 들어가는데,
agent-browser로 접근성 트리를 떠서 이메일·비밀번호 필드를 찾아 입력하니 깔끔하게 통과했다.
이어서 다단계 등록 위저드를 스텝별로 몰았다. 각 단계에서 접근성 트리를 다시 떠 다음 요소를
찾는 식으로.

그리고 **자율 QA가 실제로 이슈를 잡았다.** 스크립트 E2E로는 잘 안 드러나는 종류였다.

- **다단계 위저드의 부분 실패** — 앞 단계에서 본체가 이미 생성(201)됐는데 마지막 단계가
  실패하면, 실패했음에도 **본체 데이터는 남는다(orphan)**. 사용자는 "실패한 줄" 아는데
  실제론 데이터가 생겨 있는 상태.
- **에러가 화면에 안 뜬다** — 그 실패가 모달에는 표시되지 않고 콘솔 로그에만 남았다. 그래서
  **판정은 화면이 아니라 네트워크 응답(상태 코드)으로** 해야 한다는 교훈.
- 그 외 타이밍(선택 직후 즉시 다음 클릭 시 누락), 라벨/명칭이 화면마다 다른 지점 등.

이건 정확히 **QA가 잡았을 법한, 근데 내가 로컬에서 먼저 잡은** 이슈들이었다. 목표 달성.

### 4. 검증된 흐름을 재사용 스킬로

파일럿에서 실제로 통과시킨 흐름을, 그대로 팀에서 재사용할 수 있는 **스킬**로 박제했다.
로그인 방식, 위저드 스텝, 타이밍 대기, SPEC 대조, 데이터 정리, 그리고 파일럿에서 배운
실전 함정(orphan 정리, 네트워크로 판정 등)까지 담았다.

두 가지를 신경 썼다.

- **계정은 코드에 안 박는다.** 로그인 계정은 레포 밖 설정 파일에서 읽게 해서, 스킬을
  공유해도 자격증명은 각자 준비하게 했다.
- **팀 도구가 섞여 있다.** 팀원 중 다른 AI 코딩 도구를 쓰는 사람도 있어서, 스킬 본문이
  특정 도구에 안 묶이도록(전부 CLI 명령 기반) 하고 양쪽 도구가 읽는 위치에 함께 뒀다.

## 배운 것

1. **좋은 영상을 "봤다"로 끝내지 않기.** 개념(클로즈드 루프)을 실제 내 워크플로우의 구체적
   지점(배포 전 게이트)에 꽂아야 남는다.
2. **자율 QA는 스크립트가 못 잡는 걸 잡는다.** 특히 여러 단계·여러 계층을 걸치는 통합
   이슈(부분 실패, 에러 미표시)에서 강했다.
3. **판정은 눈이 아니라 증거로.** 화면에 에러가 안 떠도 네트워크 응답에는 남는다. "실행이
   됐다"가 PASS가 아니다.
4. **추상화(스킬화)는 검증 다음에.** 한 번 손으로 돌려 실제 함정을 확인한 뒤 스킬로 굳혔다.
   추측으로 먼저 만들었으면 틀린 워크플로우를 박제할 뻔했다.

다음은 이 게이트를 배포 직전에 자동으로 한 번 돌리는 흐름으로 다듬을 생각이다. 그렇게
"내가 먼저 잡는" 범위를 조금씩 넓혀가려 한다.
