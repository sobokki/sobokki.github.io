## 1. 왜 QA를 자동화하나 — "클로즈드 루프"

핵심 개념 하나로 시작한다. 일반적인 AI 코딩은 이렇게 흐른다.

```
요구사항 → AI가 구현 → (사람이) 검증 → 완료
                          ↑ 여기가 병목. 결국 사람이 확인해야 함
```

여기에 **AI가 스스로 하는 검증 단계**를 끼워 넣으면 흐름이 닫힌다.

```
요구사항 → 구현 → AI가 검증 → (실패면) 요구사항/구현 수정 → 다시 검증 → ... → 통과 → 완료
             └──────────────── 통과할 때까지 반복 ────────────────┘
```

이 닫힌 반복이 **클로즈드 루프(closed loop)** 다. 검증을 사람이 아니라 AI가 하게 만드는
순간, 비로소 "자율적 AI 코딩"이 된다. 그리고 이 검증을 하려면 **AI가 실제 브라우저를
조작**할 수 있어야 한다 — 그게 이 영상의 주제인 agent-browser다.

## 2. agent-browser vs Playwright MCP — 왜 더 가벼운가

브라우저를 AI가 조작하는 도구로 Playwright MCP가 이미 있는데, agent-browser는 뭐가 다를까?
차이는 **주고받는 정보의 양**이다.

| | Playwright MCP | agent-browser |
|---|---|---|
| 주고받는 것 | **DOM 트리 전체** (div, class, 태그, 속성 전부) | **접근성 트리(a11y tree)** — 의미만 |
| 컨텍스트 | 길다 → 토큰 낭비 | 짧다 → 적은 토큰으로 동일 동작 |

DOM 트리는 버튼 하나를 표현하는 데도 `div`, `class`, 온갖 속성이 다 붙는다. 반면
**접근성 트리**는 스크린리더가 쓰는 트리라, 시각 정보를 다 걷어내고 **의미(role + label)**
만 남긴다. "이건 `sign-in` 버튼" 이 두 가지면 클릭하는 데 충분하다.

그래서 같은 클릭 한 번을 하는데 agent-browser는 훨씬 적은 컨텍스트로 처리한다. AI 에이전트가
반복해서 화면을 조작할 때, 이 차이는 토큰 비용으로 그대로 쌓인다.

## 3. 동작 원리 — 4단계 준비 + 3레이어

agent-browser 내부는 이렇게 돈다.

**준비 4단계**
1. 접근성 트리 추출 (내부적으로 Playwright의 `accessibility.snapshot` 호출)
2. 인터랙티브 요소만 필터 (`onclick` 핸들러, `tabindex` 있는 것 등)
3. 각 요소에 짧은 참조 id(`ref`) 할당
4. 그 ref들을 맵으로 캐싱 (재사용 대비)

**클릭 명령이 오면**: ref 맵에서 요소를 찾아 Playwright의 `getByRole()`을 호출해 실제로 클릭.
응답은 DOM 전체가 아니라 `ref` 값만 돌아온다 → 토큰 절약.

**3레이어 구조**
```
Rust CLI (인자 파싱)  →  Node.js 데몬 (zod 검증·CDP 전달)  →  Chromium
```
클릭 명령이 오면 Rust CLI가 파싱 → 유닉스 소켓으로 Node 데몬에 전달 → 데몬이 CDP로
크로미움에 명령. 실제 데모에서는 AI가 알림신청 폼을 스스로 열고, 채우고, 제출하고,
성공 메시지까지 확인했다.

## 4. 마지막 통찰 — 에이전트 도구는 "화면"이 아니라 "텍스트"

영상이 끝에 던진 포인트가 오래 남았다. agent-browser 동작 원리를 보면 결국 **CLI**가
등장한다. AI 에이전트에게 필요한 건 복잡한 GUI가 아니라 **텍스트로 조작 가능한 인터페이스**
라는 것. 앞으로 "단순한 API + 간단한 docs"만 제공하는 서비스가 늘어날 거라는 전망이었다.

## 정리

이 영상에서 내가 챙긴 세 가지.

1. **클로즈드 루프** — 검증을 AI에게 넘기는 게 자율 코딩의 핵심. QA 자동화는 그 수단.
2. **접근성 트리로 토큰 절약** — DOM 전체 대신 의미(role+label)만. 도구 설계의 좋은 패턴.
3. **CLI-first 철학** — 에이전트용 도구는 화면보다 텍스트.

그리고 이걸 그냥 "좋은 영상"으로 넘기지 않고, 실제 내 업무 프론트엔드에 **배포 전 로컬
자율 QA**로 도입해봤다. 그 과정은 다음 글에서 정리한다.
