# Decisions: AI 개발 성장 기록 블로그

작성일: 2026-07-06

브레인스토밍 과정에서 내린 결정과 그 이유. "왜 이렇게 만들었지?"에 답하는 문서.

---

## D1. 제품이 아니라 Claude Code 스킬로 만든다

**맥락**: 이걸 mediness 제품(SPEC/WP 파이프라인)으로 만들지, 스킬로 만들지 고민.

**결정**: **전역 스킬** (`~/.claude/skills/`).

**이유**:
- 트리거("이거 블로그에 올려줘")가 Claude Code 세션 **안에서** 발생한다. Claude가 이미 대화 맥락 전체를 쥐고 있음 → 스킬이 가장 잘 하는 일.
- 1인 개인 도구지 팀 배포 제품이 아니다. mediness 제품 파이프라인은 과하다.
- 발행 대상이 기존 플랫폼(GitHub)이라 별도 백엔드 상태 관리 불필요.
- 빠른 반복 가능.

**결과**: mediness 레포는 건드리지 않는다. 스킬은 전역 위치에 둔다.

---

## D2. 블로그는 GitHub Pages + Jekyll Chirpy

**맥락**: 발행 대상 플랫폼 선택. Velog / Notion / dev.to / GitHub / 자체 제작 검토.

**결정**: **새 `sobokki.github.io` + Jekyll Chirpy 테마**.

**옵션별 비교**:

| 플랫폼 | 글쓰기 API | 판정 |
|---|---|---|
| Velog | 공식 API 없음 (비공식 GraphQL, 불안정) | ✗ |
| Tistory | Open API 2024 폐쇄 | ✗ |
| Medium | integration token 2023 중단 | ✗ |
| dev.to | 공식 REST API | △ (영문/해외용) |
| Notion | 공식 API (블로그로 보이려면 렌더러 필요) | △ |
| **GitHub Pages** | git commit = 발행 | **✓ 채택** |

**이유**:
- Claude Code가 git 네이티브 → `.md` 커밋만 하면 되니 연동 로직이 가장 단순하고 튼튼.
- 기록이 온전히 내 소유. 플랫폼 API 종료 리스크 없음.
- git 히스토리로 성장 기록이 이중 보존됨.
- 비용 0원.
- Chirpy는 한국 개발자 세팅 자료 풍부 + TIL/카테고리 구조 완비.

**기각**:
- **B안 (기존 포트폴리오 sobokkiworld에 blog 추가)**: 통합 정체성 장점 있으나 Next.js 블로그 렌더링을 직접 구현해야 해서 손이 더 감. 빠른 시작 + 개발자 블로그 표준을 우선해 Chirpy 선택.

---

## D3. 글은 한국어, 완성된 포스트 형태

**결정**: 한국어 / 바로 읽히는 블로그 글.

**이유**: 본인 기록용이고, TIL 로그보다 "완성된 글"을 원함. 나중에 TIL → 글 승격 방식도 확장 가능하되 v1은 글 단위.

---

## D4. 발행 전 검토 단계를 둔다

**결정**: 초안 → 사용자 확인 → 발행. (자동 즉시 발행 아님)

**이유**: 개인 브랜딩이 걸린 공개 글이라 오발행/품질 리스크를 막아야 함. 마찰을 최소화하되 "확인 한 번"은 유지.

---

## D5. 블로그 레포는 고정 로컬 경로

**결정**: `/Users/soeun/sobokki.github.io` (사용자 레포 관례상 홈 바로 아래).

**이유**: 스킬이 **어느 프로젝트 세션에서든** 발동하므로, 현재 작업 디렉토리(cwd)가 아니라 **고정 경로**의 블로그 레포에 써야 한다. 스킬이 이 경로를 알고 그쪽에서 커밋한다.
